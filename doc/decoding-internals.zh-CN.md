# 重建（解码）的内部机制：哪些是计算，哪些是查表

本文回答一个常见疑问：**程序是如何从剩余分片恢复出缺失数据（"剩余多项式"）的？
是人工预先算好、运行时查表吗？**

答案：都不是人工算的——而且程序里从头到尾没有出现过"多项式"这个对象。
插值在矩阵形式下等价于"解线性方程组"，恢复数据实际是**运行时做一次
高斯-约当消元求逆**。整个过程分三层，每层"算"与"查表"的分工不同。

算法背景（多项式插值 ↔ 编码矩阵的对应关系）见
[README.zh-CN.md 的"算法核心原理"](../README.zh-CN.md#算法核心原理)；
解码的逐数值数学过程（A′ 消元全程、A′⁻¹ 每行的拉格朗日身份）见
[decoding-walkthrough.zh-CN.md](decoding-walkthrough.zh-CN.md)。

---

## 第一层：域元素运算（单字节乘除）——查表，但表是编译器算的

`a * b`、`a / b` 这类 GF(2^8) 基本运算确实是查表：

- 64 KiB 完整乘法表 `MUL_TABLE`（乘法 = 一次查表）；
- `LOG_TABLE` / `EXP_TABLE`（除法 = 对数相减后查幂表）。

但这些表不是人工算好写死的，而是 `consteval` 函数在**编译期**执行
生成算法得出（`include/rse/detail/gf8_tables.hpp`）——本质上仍是程序计算，
只是把计算时刻提前到了编译时。Rust 原版同理（build.rs 在构建期生成）。

## 第二层：解码矩阵——运行时现场求逆，不是预存的

这是问题的核心。恢复缺失数据的"解方程"步骤，全部发生在 `reconstruct`
调用时：

1. **建码时**（`include/rse/reed_solomon.hpp` 的 `build_matrix`）：
   用域运算现场算出 Vandermonde 矩阵、对顶部 k×k 子阵求逆、相乘得到
   编码矩阵 A——运行时计算；
2. **重建时**（同文件 `get_data_decode_matrix`）：根据哪些分片还存活，
   从 A 中抽出对应的 k 行拼成方阵 A′，调用 `include/rse/matrix.hpp` 的
   `invert()` → `gaussian_elim()` 做**高斯-约当消元**，当场求出 A′⁻¹。

**为什么不能预先算好存表？** 因为 A′ 取决于*具体哪几个分片丢了*，
丢失组合有 C(n, m) 种——例如 20+10 配置约有三千万种组合，预存不现实。
只能等故障发生、确定缺失模式后现算。

不过有一个折中的"运行时记忆化"：求出的逆矩阵按**缺失下标列表**为键
放入 LRU 缓存（容量 254，`DATA_DECODE_MATRIX_CACHE_CAPACITY`）。
同一缺失模式第二次出现时就真的变成查表了。这在实际场景中极为常见——
某块盘坏了，后续成千上万个条带的重建都是同一缺失模式，
只有第一个条带付出 O(k³) 的求逆代价。

## 第三层：真正恢复数据——矩阵乘向量，SIMD 批量运算

拿到 A′⁻¹ 后：

```
缺失分片 = A′⁻¹ 的对应行 · 现存分片
```

落到实现上是对每条切片做 `mul_slice_xor`（第一层的查表乘法，
SIMD 加速，见 README 的"SIMD 如何加速 RS 算法"一章）。
这一层占数据量的大头，但每字节只是乘加，没有任何"求解"过程。

数据分片全部恢复后，缺失的校验分片按正常编码流程重算。

---

## 总结

| 层次 | 做什么 | 计算时机 | 形式 |
| --- | --- | --- | --- |
| 域元素 | 单字节乘/除 | **编译期**生成表 | 运行时查表 |
| 解码矩阵 | 高斯-约当消元求 A′⁻¹ | **运行时**（按缺失模式） | 现算 + LRU 缓存记忆化 |
| 数据恢复 | A′⁻¹ × 现存分片 | **运行时**（每字节） | SIMD 批量乘加 |

插值/解方程的"智力活"（求逆）在运行时由高斯消元现算、按缺失模式缓存；
只有最底层的单字节域乘法是查表，而那张表也是编译器算出来的——
整条链路没有任何人工预计算。
